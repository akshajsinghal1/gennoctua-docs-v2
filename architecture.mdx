## Two Auth Paths

### Path A — Frontend-Only (via Gennoctua)

Used by brands with no backend (Shopify, Webflow, plain HTML).

```
1. Brand receives gn_pk_... key from Gennoctua
2. SDK calls POST /auth/token with publicKey
3. Gennoctua validates key → returns 1hr JWT
4. SDK uses JWT to call /api/tryon/submit and /api/tryon/status
5. Gennoctua looks up brand's HP API key → proxies to HP
```

```
Browser SDK
    │
    ├─── POST /auth/token { publicKey: "gn_pk_..." }
    │         ↓
    │    Gennoctua Backend
    │         │ validates brand key in SQLite DB
    │         └─── returns { token: "eyJ...", expiresIn: 3600 }
    │
    ├─── POST /api/tryon/submit (Bearer token)
    │         ↓
    │    Gennoctua Backend
    │         │ validates JWT, looks up hp_api_key
    │         └─── forwards to HP (X-API-Key: hpk_...)
    │
    └─── GET /api/tryon/status/:jobId (Bearer token)
              ↓
         Gennoctua Backend
              │ validates JWT, looks up hp_api_key
              └─── streams SSE from HP back to browser
```

### Path B — Full-Stack (brand's own backend)

Used by brands that have their own Node.js, Next.js, Express, etc. backend.

```
1. Brand stores HP API key in their .env
2. Brand builds token endpoint (POST /auth/tryon-token)
3. Brand builds proxy endpoints (POST + GET /api/tryon/...)
4. SDK calls brand's own backend with JWT
5. Brand's backend forwards to HP
```

```
Browser SDK
    │
    ├─── POST /auth/tryon-token
    │         ↓
    │    Brand's Backend
    │         │ generates JWT with brand's own JWT_SECRET
    │         └─── returns { token: "eyJ...", expiresIn: 3600 }
    │
    ├─── POST /api/tryon/submit (Bearer token)
    │         ↓
    │    Brand's Backend
    │         │ validates JWT, uses HP_API_KEY from .env
    │         └─── forwards to HP
    │
    └─── GET /api/tryon/status/:jobId (Bearer token)
              ↓
         Brand's Backend → streams SSE from HP
```

## What Runs Where

| Component | Where it runs | What it does |
|---|---|---|
| Face detection | **Browser** (WebAssembly) | Detects faces in uploaded photos |
| Pose scoring | **Browser** (WebAssembly) | Scores body poses for best photo selection |
| Photo selection | **Browser** | Picks best male/female/child profile photo |
| Token generation | **Backend** (Gennoctua or brand) | Issues short-lived JWTs |
| Job submission | **Backend** (proxy) | Forwards multipart form to HP |
| SSE streaming | **Backend** (proxy) | Streams job status back to browser |
| Result caching | **Browser** (IndexedDB) | Caches try-on results for 7 days |
| AI generation | **HP** | Generates the try-on image |

## Data Flow: Full Try-On Lifecycle

```
1. sdk.init()
   └── Loads face-api.js + MediaPipe models (CDN, ~5-10s first time)
   └── Fetches JWT token from auth endpoint

2. sdk.ingestImages(files)
   ├── Filters valid image files (max 80)
   ├── Runs face detection on each photo
   ├── Classifies gender + age per face
   ├── Scores pose quality per full-body photo
   ├── Selects best male + female candidate
   └── Returns SelectionSummary

3. sdk.personalize({ imageUrl, productType })
   ├── Checks result cache (IndexedDB) → cache hit? return instantly
   ├── Checks active job cache → restoring? skip submission
   ├── Checks rate limiter → too fast? throw RATE_LIMITED_CLIENT
   ├── Submits job (POST /submit) → receives job_id
   ├── Opens SSE stream (GET /status/:jobId)
   ├── Polls until COMPLETED or FAILED (every 1.5s, max 3 min)
   ├── Caches result in IndexedDB (7 day TTL)
   └── Returns PersonalizationResult { imageUrl, cacheHit, jobId }
```

## HP API

HP is the AI backend that generates the try-on images. Gennoctua manages the relationship with HP — brands never interact with it directly.

| Endpoint | Method | Purpose |
|---|---|---|
| `/api/tryon/submit` | POST | Submit a try-on job |
| `/api/tryon/status/:jobId` | GET (SSE) | Stream job status |
| `/api/eyewear/submit` | POST | Eyewear category |
| `/api/eyewear/status/:jobId` | GET (SSE) | Eyewear status |
| `/api/footwear/submit` | POST | Footwear category |
| `/api/makeup/submit` | POST | Makeup category |
| `/api/accessories/submit` | POST | Accessories category |

The SDK automatically routes to the correct endpoint based on `productType`.

## Security Model

| Threat | Mitigation |
|---|---|
| HP API key exposed in browser | Key lives on backend only, never sent to browser |
| Stolen JWT used to make requests | JWTs expire in 1hr, rate limiting caps abuse |
| Brand key leaked | Only gets JWTs, cannot directly call HP |
| Excessive requests (abuse) | Client-side + server-side rate limiting |
| Expired token | SDK auto-refreshes in PublicKey mode, `getToken` called again in proxy mode |
