## Principles

The SDK is designed around three security principles:

1. **API keys never touch the browser** — HP API keys live on the backend only
2. **Short-lived tokens** — browsers only ever hold JWTs that expire in 1 hour
3. **Defense in depth** — rate limiting, token validation, and CORS work together

---

## What Runs Where

| Data | Location | Why |
|---|---|---|
| HP API key | Backend only | If exposed in browser, anyone could use it |
| JWT session token | Browser (memory) | Short-lived (1hr), limited to try-on only |
| Brand key (`gn_pk_...`) | Browser (code) | Safe — only fetches JWTs, can't call HP directly |
| User photos | Browser only | Never stored on any server (processed locally) |
| Try-on result URLs | Browser IndexedDB | Cached for 7 days, local to device |

---

## Authentication Flow

See [Authentication](/security/authentication) for the full JWT lifecycle.

**Short version:**
```
1. Brand key → Gennoctua → 1hr JWT
2. JWT → proxy request headers
3. Backend validates JWT → forwards with HP API key
4. HP API key never reaches the browser
```

---

## User Photo Privacy

Photos uploaded by the shopper are processed **entirely in the browser**:

- Face detection runs locally (WebAssembly via face-api.js)
- Pose scoring runs locally (WebAssembly via MediaPipe)
- The selected profile photo is sent to the AI backend **only** when Try On is clicked
- No photos are stored on any Gennoctua server
- Photos are not logged, indexed, or retained

<Info>
Only the **selected profile photo** (one photo) is sent to HP for AI generation. All other uploaded photos are processed and discarded in the browser.
</Info>

## Token Security

### JWT Contents

JWTs issued by Gennoctua contain:

```json
{
  "brandKey": "gn_pk_mauijim_xxx",
  "brandName": "Maui Jim",
  "iat": 1234567890,
  "exp": 1234571490
}
```

They do **not** contain:
- HP API keys
- User identifiers
- Personal data

### Token Lifetime

Tokens expire after **1 hour**. A stolen token can only:
- Submit try-on jobs (costs against that brand's HP quota)
- Poll for job status

Rate limiting further caps how many jobs can be submitted.

### Token Refresh

**Frontend-only mode:** The SDK automatically fetches a new token from Gennoctua when the current one expires. No action needed.

**Full-stack mode:** The SDK calls your `getToken()` function before each request. If the token is expired, your function should fetch a fresh one.

---

## Rate Limiting as Abuse Prevention

Even if a brand key or token is leaked, rate limiting prevents runaway abuse:

| Limit | Default |
|---|---|
| Cooldown between requests | 10 seconds |
| Max try-ons per session | 20 |
| Max try-ons per product | 3 |

Server-side limits are also enforced by the Gennoctua backend per brand.

---

## HTTPS Requirement

The SDK requires HTTPS in production because:

1. Browser AI models (WebAssembly) require a secure context
2. CORS policies block cross-origin requests from non-HTTPS pages
3. JWT tokens must not be transmitted over unencrypted connections

**Local development** on `localhost` or `127.0.0.1` is exempt from HTTPS requirements.

---

## CORS

In full-stack mode, your backend must allow CORS from your frontend origin:

```js
// Express
app.use(cors({
  origin: "https://yourbrand.com",
  methods: ["GET", "POST"],
}));
```

In frontend-only mode, the Gennoctua backend handles CORS automatically.

---

## Brand Key Safety

Brand keys (`gn_pk_...`) are designed to be safe in frontend code:

- They can only be used to fetch short-lived JWTs
- They cannot directly call HP
- If a brand key is leaked, you can deactivate it instantly via the Admin API
- Rate limiting caps any abuse

<Warning>
Brand keys should still be treated as sensitive. Don't commit them to public GitHub repos. Use environment variables.
</Warning>
